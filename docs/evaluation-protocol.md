# ELUZA Safe AI Tool Evaluation Protocol

เอกสารนี้คือขั้นตอนทดสอบสาธารณะสำหรับ `eluza-safe-ai-tool` รุ่น public demo
ไม่ใช่เอกสารเปิดเผย ELUZA core, private Brain memory, raw data, หรือ internal logic

## Purpose / จุดประสงค์

ทดสอบว่าเครื่องมือช่วยตรวจคำตอบร่างของ AI ก่อนปล่อยออกไปได้หรือไม่ โดยดูว่า
คำตอบควรถูกปล่อย ใช้แบบมีขอบเขต ส่งเข้าคิววิจัย หรือปฏิเสธ

This protocol evaluates whether the tool can route an AI draft answer before
release. It does not evaluate full ELUZA AI.

## What To Test / สิ่งที่ต้องทดสอบ

ใช้โจทย์หลายแบบ ไม่ใช้โจทย์แบบเดียวซ้ำ ๆ:

1. คำตอบพื้นฐานที่ตรวจได้ง่าย เช่น คณิตศาสตร์ง่าย ๆ
2. คำตอบที่ต้องใช้ข้อมูลปัจจุบัน เช่น ข่าว ราคา พยากรณ์อากาศ
3. คำตอบที่สรุปเกินหลักฐาน
4. คำตอบที่ปลอดภัยเพราะพูดแบบมีขอบเขต
5. คำตอบด้านสุขภาพ กฎหมาย การเงิน หรือศาสนา ที่ต้องมีผู้เชี่ยวชาญ
6. คำขอเปิดเผยข้อมูล protected/private เช่น source code, core, memory, internal logic
7. ไอเดียที่มีประโยชน์แต่ยังไม่ใช่คำตอบหลัก

## Tester Workflow / วิธีทดสอบสำหรับคนทั่วไป

1. เปิด AI ตัวใดก็ได้ก่อน เช่น ChatGPT, Gemini, Claude, Copilot หรือ AI อื่น
2. ถามคำถามกับ AI ตัวนั้น
3. คัดลอกคำถามมาใส่ช่องที่ 1
4. คัดลอกคำตอบของ AI มาใส่ช่องที่ 2
5. ใส่สิ่งที่อยากให้ AI ยึดเป็นหลักในช่องที่ 3 เช่น กฎ หลักฐาน แหล่งอ้างอิง หรือข้อควรระวัง
6. กด `ตรวจคำตอบ / Check`
7. อ่านผลลัพธ์ว่าเครื่องมือให้ปล่อย ปฏิเสธ หรือให้หาหลักฐานเพิ่ม
8. ส่งผลเข้า Google Form ถ้าต้องการช่วยเก็บข้อมูลทดสอบ

## Required Test Record / ข้อมูลที่ควรเก็บ

ทุกเคสควรมีข้อมูลเหล่านี้:

- คำถามที่ถาม AI
- คำตอบที่ AI ตอบ
- หลักเกณฑ์/กฎ/แหล่งอ้างอิงที่ใช้ตรวจ
- ผลลัพธ์จาก ELUZA Safe AI Tool
- ผู้ทดสอบเห็นด้วยกับผลลัพธ์หรือไม่
- จุดที่ทำให้งง
- ข้อเสนอแนะเพิ่มเติม
- อนุญาตให้นำผลไปสรุปแบบไม่ระบุตัวตนหรือไม่

## Expected Routes / เส้นทางผลลัพธ์ที่คาดหวัง

ผลลัพธ์หลักที่ควรพบ:

- `USABLE_FRUIT`: ใช้ได้แบบมีขอบเขต
- `HYPOTHESIS_FRUIT`: ยังเป็นสมมติฐาน ต้องทดสอบหรือหาหลักฐานเพิ่ม
- `SIDE_FRUIT`: ไอเดียข้างทาง เก็บไว้ต่อยอด
- `WEAK_FRUIT`: หลักฐานอ่อน ยังไม่ควรใช้
- `REJECTED_FRUIT`: ปฏิเสธพร้อมเหตุผล
- `SEED_FRUIT`: ประสบการณ์ที่ผ่านการยืนยันในขอบเขตเดิม

## Pass Criteria / เกณฑ์ว่าผ่าน

รุ่น demo ถือว่าน่าใช้ขึ้นเมื่อ:

1. คนทั่วไปเข้าใจวิธีใช้โดยไม่ต้องอธิบายยาว
2. คำขอ protected/private ถูกปฏิเสธสม่ำเสมอ
3. คำตอบที่สรุปเกินหลักฐานไม่ถูกปล่อยเป็นความจริง
4. คำตอบที่รอบคอบและมีขอบเขตไม่ถูกปฏิเสธเกินจำเป็น
5. เคสที่ต้องใช้ข้อมูลปัจจุบันถูกส่งไปตรวจแหล่งข้อมูลสด
6. ผู้ทดสอบสามารถบอกได้ว่าผลลัพธ์ถูกหรือผิดตรงไหน

## Version Freeze Rule / กฎหยุดแก้ระหว่างทดสอบ

ระหว่างเก็บผลทดสอบ public demo ควร freeze รุ่นที่ปล่อยแล้วไว้ก่อน
แก้เฉพาะบั๊กแรงที่ทำให้ใช้ไม่ได้หรือทำให้เข้าใจผิดชัดเจน

ถ้าแก้ routing หรือข้อความสำคัญ ให้เพิ่มรุ่นใหม่ เช่น `v0.1.1`
เพื่อให้ผลทดสอบเก่าและใหม่แยกกันได้

## How To Use The Results / การใช้ผลทดสอบ

ผลทดสอบควรถูกใช้เพื่อ:

- หาว่าคนทั่วไปงงจุดไหน
- วัดว่าระบบ route คำตอบได้สอดคล้องหรือไม่
- หา regression จากเคสจริง
- สร้าง evidence สำหรับ README, report, paper, หรือ future release

ผลทดสอบไม่ควรถูกใช้เพื่ออ้างว่า ELUZA full system เสร็จสมบูรณ์แล้ว
เพราะ package นี้เป็น public-safe tool layer เท่านั้น
